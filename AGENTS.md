# AGENTS.md

이 문서는 Exportra 저장소에서 Codex와 팀원이 같은 기준으로 작업하기 위한 운영 원칙을 정리합니다.

## 프로젝트 목적

Exportra는 공공 무역·산업 데이터와 AI를 활용하여 제조기업의 해외시장 진출 의사결정을 돕는 AI Export Copilot입니다.

이 프로젝트는 단순한 일회성 데모가 아니라, 실제로 확장 가능한 오픈소스 제품처럼 개발하는 것을 목표로 합니다.

## MVP 우선순위

초기 MVP는 작지만 끝까지 동작하는 하나의 사용자 흐름에 집중합니다.

우선순위는 다음과 같습니다.

1. 기업 및 제품 정보 입력
2. 분석 요청
3. HS Code 후보 추천
4. 유망 수출국 TOP3 추천
5. 국가별 시장성, 리스크, 인증, 지원사업, 실행 로드맵 제공
6. 공공데이터 기반 정보와 AI 생성 설명 구분
7. 팀원이 로컬에서 실행 가능한 구조 유지

## Codex 작업 원칙

Codex는 단순 코드 생성기가 아니라 AI Tech Lead 역할을 맡습니다.

Codex는 작업할 때 다음 요소를 함께 고려해야 합니다.

- 프로젝트 일정
- MVP 범위
- 기술 선택의 단순성
- 초보자도 이해 가능한 구조
- 문서 품질
- 오픈소스 프로젝트로서의 유지보수성
- 공공데이터와 AI 활용의 신뢰성

## 파일 수정 전 계획 제시 원칙

Codex는 파일을 생성하거나 수정하기 전에 먼저 계획을 제시해야 합니다.

계획에는 다음 내용을 포함합니다.

- 어떤 파일을 수정할지
- 왜 수정이 필요한지
- 수정 후 기대 결과가 무엇인지
- README, docs, frontend, backend, 설정 파일 중 어디에 영향을 주는지

사용자가 승인하기 전에는 파일을 생성하거나 수정하지 않습니다.

## 파일 수정 규칙

한 번에 너무 많은 파일을 수정하지 않습니다.

기본 원칙은 다음과 같습니다.

- 한 단계에 하나의 문서
- 한 단계에 하나의 기능
- 한 단계에 하나의 설정 변경
- 관련 없는 리팩터링 금지
- 사용자 또는 팀원이 수정한 내용을 임의로 되돌리지 않기

여러 파일을 함께 수정해야 하는 경우, 먼저 이유를 설명하고 승인받습니다.

## 초보자 팀 기준 코드 작성 원칙

대부분의 팀원이 풀스택 개발 경험이 많지 않다는 점을 기준으로 작업합니다.

코드는 다음 기준을 따릅니다.

- 똑똑한 코드보다 읽기 쉬운 코드 우선
- 복잡한 추상화보다 명확한 구조 우선
- 로컬에서 실행하기 쉬운 구조 우선
- 역할이 분명한 폴더와 파일 구조 유지
- 필요한 경우에만 짧고 의미 있는 주석 작성
- 익숙한 패턴을 먼저 사용하고, 고급 구조는 필요할 때만 도입

MVP 단계에서는 불필요하게 복잡한 구조를 피합니다.

## API 키와 민감정보 관리

API 키, 비밀번호, 토큰, 개인 설정 파일은 절대 커밋하지 않습니다.

민감정보는 환경변수로 관리합니다.

예시는 `.env.example`에 작성할 수 있지만, 실제 값은 `.env` 같은 로컬 파일에만 보관합니다.

민감정보 예시는 다음과 같습니다.

- `OPENAI_API_KEY`
- 데이터베이스 사용자명과 비밀번호
- 공공데이터 API 서비스 키
- 배포 서버 접속 정보
- GitHub 토큰

## 문서 업데이트 원칙

기능, 실행 방법, API, 데이터 활용 방식이 바뀌면 관련 문서도 함께 업데이트합니다.

중요 문서는 다음과 같습니다.

- `README.md`
- `docs/product/DATA_SOURCES.md`
- `docs/product/USER_FLOW.md`
- `docs/product/SCREEN_SPEC.md`
- `docs/architecture/API_SPEC.md`
- `docs/architecture/DB_SCHEMA.md`
- `docs/architecture/SYSTEM_ARCHITECTURE.md`

`README.md`는 범용 오픈소스 프로젝트 소개 문서로 유지합니다.

특정 공모전 전략은 README에 직접 강조하지 않고 `docs/contests/` 아래에서 관리합니다.

## MVP에서 제외할 기능

다음 기능은 MVP 단계에서 제외합니다.

명시적인 승인 없이 구현하지 않습니다.

- 회원가입/로그인
- 결제
- 관리자 페이지
- 복잡한 권한 관리
- 실시간 크롤링
- 모바일 앱
- 다국어 지원
- Redis
- Kubernetes
- MSA
- OAuth
- 복잡한 CI/CD
- 대규모 데이터 파이프라인
- 실제 수출 계약 매칭
- 실제 지원사업 신청 연동

## 완료 기준

작업은 다음 조건을 만족할 때 완료된 것으로 봅니다.

- 요청한 변경이 구현 또는 문서화됨
- MVP 범위를 벗어나지 않음
- 민감정보가 커밋되지 않음
- 필요한 경우 관련 문서가 함께 업데이트됨
- 초보자 팀원이 이해할 수 있는 구조를 유지함
- 변경 내용을 팀원에게 설명할 수 있음

## 커밋 전 체크리스트

커밋 전에는 다음을 확인합니다.

- 하나의 커밋에 관련 있는 변경만 포함되었는가?
- 커밋 메시지가 Conventional Commit 형식을 따르는가?
- API 키나 민감정보가 포함되지 않았는가?
- 로컬 전용 파일이나 생성물이 제외되었는가?
- 필요한 문서가 함께 업데이트되었는가?
- 커밋 후 작업 트리가 깨끗한가?

권장 커밋 메시지 예시는 다음과 같습니다.

- `docs(readme): introduce project overview`
- `docs(data): document public data strategy`
- `docs(agents): define repository working rules`
- `chore(config): add gitignore`
- `chore(env): add environment variable example`
